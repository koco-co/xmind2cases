# 文档维护指南

## 原则

### Keep It Simple (KIS)
- 文档应当简洁明了，避免过度复杂化
- 使用简单的语言和结构
- 聚焦于如何使用，而不是实现细节
- 每个文档应该有明确的单一目标

### 同步更新
- 代码和文档必须保持同步
- 任何 API 变更必须同时更新文档
- 新功能添加时必须包含文档
- 删除功能时必须移除相关文档

### 版本一致
- 文档版本与软件版本保持一致
- 更新文档时更新版本号
- 在文档中明确标注版本信息
- 重大变更时考虑创建版本迁移指南

## 检查清单

### 发布前检查
- [ ] 所有新功能都有对应的文档
- [ ] API 变更已更新文档
- [ ] 示例代码与实际实现一致
- [ ] 文档没有拼写错误
- [ ] 链接都有效且指向正确的目标
- [ ] 文档格式统一且美观
- [ ] README.md 更新到最新版本
- [ ] 安装说明准确无误
- [ ] 快速入门指南完整可用

### 提交 PR 时检查
- [ ] 文档变更已包含在提交中
- [ ] 所有新增文档都经过拼写检查
- [ ] 代码示例已测试并通过
- [ ] 文档没有泄露敏感信息
- [ ] 外部链接使用 HTTPS 协议
- [ ] 图片和资源路径正确
- [ ] 文档结构符合项目规范
- [ ] 版本号已更新（如有需要）

## 自动检查说明

### pre-commit 钩子
项目配置了 pre-commit 钩子，会在提交时自动执行以下检查：
- 文档拼写检查（如果有拼写检查工具）
- Markdown 语法验证
- 链接有效性检查（部分检查）

### 文档格式检查
使用以下工具进行文档格式检查：

```bash
# 检查 Markdown 语法
python scripts/check_docs.py

# 或者使用 markdownlint
npm install -g markdownlint
markdownlint docs/
```

### 文档一致性检查
- 确保所有文档使用相同的标题层级
- 代码块语法高亮一致
- 文件命名规范统一
- 目录结构符合逻辑

### 版本控制
- 每次重大变更都应更新文档版本
- 使用语义化版本号（如 v1.2.0）
- 在文档头部添加版本信息
- 重大变更考虑添加变更日志

## 最佳实践

### 文档组织
- 按功能模块组织文档
- 使用清晰的目录结构
- 提供导航和索引
- 保持文件大小合理（建议 <500 行）

### 写作风格
- 使用主动语态
- 保持句子简洁
- 提供具体的示例
- 避免技术术语堆砌

### 维护流程
1. 代码变更时立即更新相关文档
2. 定期审查和更新过时文档
3. 收集用户反馈并改进文档
4. 保持文档的准确性和时效性

## 贡献指南

### 新文档添加
1. 在合适的位置创建文档文件
2. 遵循项目文档模板
3. 添加必要的目录导航
4. 提交时使用合适的提交消息

### 现有文档更新
1. 确保更新符合最新代码
2. 保持文档风格一致
3. 更新版本信息（如有需要）
4. 测试所有示例代码

## 工具推荐

### 文档生成
- **MkDocs**: 静态站点生成
- **GitBook**: 交互式文档
- **Docusaurus**: React 文档站点
- **VitePress**: Vue 文档站点

### 拼写检查
- **cspell**: 多语言拼写检查
- **write-good**: 英语写作质量检查
- **proselint**: 英语风格检查

### 链接检查
- **markdown-link-check**: Markdown 链接检查
- **linkchecker**: 网站链接检查
- **lychee**: 链接验证工具

记住：好的文档是项目的成功关键，保持文档的准确性和完整性对用户体验至关重要。
